webmeetInfra Documentation

webmeetInfra is the Ploinky repository for the WebMeet media runtime. It ships two agents: liveKitServerAgent, which supervises Redis, LiveKit Server, and LiveKit Egress inside one container, and turnServerAgent, which runs Coturn as a dedicated TURN/STUN relay with shared-secret REST/ephemeral auth and a fail-closed peer ACL. Neither agent runs Nginx or Certbot; the WebMeet signaling edge (TLS termination, reverse proxying) lives in basic/web-publishing, a different repository.

The liveKitServerAgent manifest pulls docker.io/assistos/livekit-server-agent:webmeet-infra by default. The image is published through the manual publish-livekit-server-agent.yml workflow in AssistOS-AI/container-image-builds, which checks out this repository as its build context, uses the centralized Dockerfile from container-image-builds/images/livekit-server-agent/Dockerfile, publishes linux/amd64 and linux/arm64 variants, includes Node 24 plus the Ploinky dependency-cache toolchain and snapshot-pinned libc-bin/getent startup dependency, and authenticates against Docker Hub using that repository's DOCKERHUB_TOKEN secret; the token value must never be stored in any repository. The workflow retains stable tags while validating, exposing, and reporting the pushed multiarchitecture manifest's immutable sha256 reference. turnServerAgent pulls the upstream docker.io/coturn/coturn image directly, pinned by digest, with no custom build or publishing workflow.

LiveKit API credentials and the shared TURN secret are host-hook-only manifest env (runtime: false): Ploinky exposes them to trusted preinstall hooks but omits them from container environment metadata. LiveKit preinstall writes owner-only generated config. TURN preinstall writes one atomically replaced secret file behind a mode-0700 host parent and mounts its readable leaf directory read-only for Coturn's fixed non-root UID; Coturn builds the full mode-0600 config only inside its private runtime directory.

The schema-2 topology has two explicit trust zones. liveKitServerAgent attaches to primary webmeet-signaling and secondary webmeet-turn; turnServerAgent attaches only to primary webmeet-turn. Profiles inherit these root network blocks atomically. Manifests declare no aliases; Ploinky derives livekitserveragent and turnserveragent from canonical agent IDs. In default/dev, only TURN's listener and relay range are host-published on 127.0.0.1; LiveKit media is relay-only. In prod, LiveKit publishes only its media ports, while signaling, Redis, Egress health, and both agents' readiness checks remain private. Production TURN requires a public multi-label TLS hostname; quota/rate/lifetime inputs are bounded at host and container boundaries, shared TURN auth cannot be operator-overridden, and storage/TLS paths reject symlink escapes. Neither agent publishes a health port; each is a start-only service using a root-level health.readiness.script, rather than an MCP agent.

The DS specifications are the source of truth for contracts, runtime assumptions, and Ploinky invariants. Start with DS000-vision.md, DS001-coding-style.md, DS002-livekit-server-agent.md, and DS004-turn-server-agent.md before changing either agent's runtime.
